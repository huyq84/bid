# 百草园周报填报平台 — 方案设计 v5

> 版本: v5
> 日期: 2026-06-05
> 上游: v3 方案 (v3 已敲定的部分保留) + v4 工作流细化
> 性质: **不受 v3 框架约束的重新设计**，按用户最终要的结果反推
> 关系: v3 是历史方案, v4 是过渡细化, **v5 是当前主线**
> 阅读建议: 先看 §0 立场, 然后看 §3 数据模型, 其他章节按需

## 0. 立场 (v5 核心)

v5 与 v3/v4 的 3 个本质差异:

### 0.1 不受 v3 框架约束

v3 是在 "5.18 那期周报的人工填报" 基础上设计的, 保留了大量当时的具体取值 (5 个场地、21 行甘特、ECC 91.82% 等). v5 把这些都视为 **数据**, 不视为 **配置**.

具体:
- 5.18 那期场地数 = 5 (食堂/高管/北咖啡/南咖啡/南健身) — v5 不硬编码
- 09 周计划 21 行 — v5 不硬编码 (行数 = 当期用户新增的行数)
- "4.2 高管区" 这个副标题 — v5 不硬编码 (前缀 + 实例数据动态组合)
- 副标题前缀 (4.2 / 5.1) — v5 不硬编码 (侧边栏树自动编号)

### 0.2 数据反推版式 (data-driven templates)

v4 的版式定义里写了 `repeatBy: "area"` + `repeatValues: ["高管区", "食堂区", ...]`, 这是 **配置型**. v5 改成:

```
版式定义里只写 "这是个可重复版式, 重复键 = area"
实例数 = scan(数据) 后的 group 数
如果某期数据只有 3 个 area, 实例就是 3 个, 不是 5 个
```

版式本身只规定 "有几类 + 怎么 repeat + 数据形状", 实例化由数据反推.

### 0.3 左侧边栏是可编辑树

v3/v4 的侧边栏是 12 个固定按钮. v5 改成:

- **树结构**, 节点可增 / 改 / 删 / 拖拽 (位置 + 层级)
- **自动前缀**: 新增一级节点 → 前缀自动 "第 N 章", N = 当前同级最大 + 1
- **自动子级前缀**: 一级节点下的子节点前缀 "N.M", N = 父节点前缀序号, M = 父节点下同级最大 + 1
- **前缀可改但格式可设**: 在 "设置" 页面配默认前缀格式 (e.g. "第{N}章" / "Chapter {N}" / "({N})")
- **前缀自动注入到副标题**: 副标题 = 侧边栏节点的 prefix + 空格 + (实例动态数据) (可选)
- **前缀不进入一级标题渲染**: 一级标题 = 节点名, 不含 prefix; prefix 只用于侧边栏显示 + 副标题注入

## 1. 13 张版式重新分类

按 "数据反推" 原则, 13 张版式分 3 大类:

### 1.1 固定型 (single) — 实例数恒为 1

- 01 封面
- 02 目录
- 03 组织架构 (花名册是单张大表, 不按场地拆)
- 0301 重要节点一览 (单张时间线)
- 06 人员统计 (单张大表)
- 07 ECC 销项 (单张大表)
- 08 图纸深化 (单张大表)
- 09 周计划 (单张按钮矩阵大表)
- 11 施工段计划 (单张大表)
- 12 协调事宜 (单张大表)

### 1.2 重复型 (repeatable) — 实例数 = 数据 group 数

- 04 上周完成: 重复键 = area
- 05 工作现场: 重复键 = area
- 10 施工段划分: 重复键 = floor (或 section)

### 1.3 自由型 (free) — 随用户选择出现

v5 引入 2 个新的自由型版式:

- 13 随手拍 (v5 新增) — 每天随手拍的照片池, 周报时按 area / floor 自由选用
- 14 封尾 (v5 预留) — 用户首次添加 "封尾" 节点后, 自动加载致谢模版, 可编辑可恢复

**注**: 13/14 是 v5 提出的新章节, v3 13 个章节里没有. v5 把章节数从 13 扩到 14 (含封尾), 用户可自行决定要不要加.

## 2. 数据反推实例化算法

### 2.1 通用反推规则

```js
// 伪代码
function materializeInstances(template, data) {
  if (template.type === 'single') {
    return [{ id: template.id, data: data, isDefault: true }];
  }
  if (template.type === 'repeatable') {
    const key = template.repeatBy;  // 'area' | 'floor' | 'section'
    const groups = groupBy(data.rows || data.photos || data.images, key);
    return Object.entries(groups).map(([k, rows]) => ({
      id: `${template.id}-${slugify(k)}`,  // e.g. "04-gaoguan"
      keyValue: k,                          // 实例取值
      data: rows
    }));
  }
  if (template.type === 'free') {
    return data.items || [];  // 用户自由添加的实例
  }
}
```

### 2.2 兜底

- **重复型版式, 数据为空**: 实例 = 1 个空实例 (不是 0 个), 让用户能填
- **重复键的值含特殊字符**: slugify 处理 (中文 → 拼音首字母, 见 §3.5)
- **跨期一致性**: 上期 04 有 5 个 area, 本期用户加了 1 个新 area "南门卫室". 本期 04 实例 = 6 个, 旧 5 个 area 的任务文本继承 (per v4 §1.3 策略表), 新 1 个 area 空实例

### 2.3 跨期实例同步

v4 §1.3 "从 Wxx-1 复制" 策略表 v5 沿用, 但加一条:

- 重复型版式的实例数 = max(上期实例数, 本期数据 group 数, 1)
- 也就是说, 上期 5 个 area, 本期用户填了 3 个, **不会**自动删掉 2 个空实例. 本期 5 个 area, 上期 3 个, **会**补 2 个新空实例

### 3.2 关键设计点

- **nav_tree 是真相源**: 侧边栏展示什么、副标题怎么拼、目录怎么生成, 都从这个树读
- **pages_data 是数据源**: 实际填报内容
- **page_id 与 HTML 文件名 1:1 对应**: page_id="04" 渲染 `preview_pages/04.html`
- **prefix 是计算字段**: 树结构变化时 (增/删/拖拽), 自动重算所有节点的 prefix
- **prefix 不进一级标题**: `一级标题` = `node.name`, 不含 prefix

### 3.3 副标题拼装规则 (v5 关键)

```
副标题 = node.prefix (可能为空) + " " + (实例动态数据) (可选)

例 1: 04 的 "gaoguan" 实例
  node.prefix = "2.1"
  instance.keyValue = "高管区"
  副标题 = "2.1 高管区"

例 2: 01 封面
  node.prefix = null
  副标题 = "" (封面无副标题)

例 3: 12 协调事宜
  node.prefix = "4.1"
  instance 不存在 (单实例)
  副标题 = "4.1"
```

### 3.4 prefix_format 配置

```jsonc
"prefix_format": {
  "level1": "第{N}章",     // 默认 "第1章", "第2章"...
  "level2": "{N}.{M}",     // 默认 "1.1", "1.2", "2.1"...
  "closing": ""            // 封尾不自动编号
}
```

用户可在 "设置" 改: e.g. `level1: "Chapter {N}"` → 一级前缀变 "Chapter 1", "Chapter 2"...

### 3.5 实例 id 的 slugify

中文 area 值 → 实例 id 后缀. 规则:
- "高管区" → "gaoguan" (拼音首字母, 查表)
- "食堂区" → "shitang"
- "北塔咖啡厅" → "beita-kafeiting"
- v1 用查表 (常见 20 个 area 维护一张表), 查不到 fallback 到 `area-{hash}`

- 节点的 name / prefix **不继承** (prefix 要按本期同级最大重算)

### 4.2 手动模式 vs 按钮模式 (Q1 回答)

每行有一个 **手动模式开关按钮** (默认关):

- **关闭 (默认, 按钮模式)**: 7 天格子可点击; "工作天数" 实时统计
- **打开 (手动模式)**: 7 天格子 **置灰不可点**; 之前已点亮的格子 **自动清除**; "工作天数" 变为可填数字框; 弹 toast 提示 "已开启手动模式, 之前选择的天数已清空"

UI 状态:
- 关闭: 格子 = 蓝色按钮; 天数列 = 灰底只读
- 打开: 格子 = 灰底不可点; 天数列 = 白底可填

切换时:
```js

}
```

## 5. 左侧边栏树结构 (Q1 回答 §1)

### 5.1 节点类型

```ts
type NavNode = {
  id: string;                              // 唯一 id, "n01" / "n0401" / 自定义
  type: 'page' | 'chapter';                // page = 关联到一个 page_id; chapter = 一级标题 (无 page_id)
  page_id?: string;                        // 当 type='page' 时必填, 关联到 pages_data
  name: string;                            // 显示名 (一级标题 / 二级标题)
  prefix?: string;                         // 自动编号, 可改
  parent_prefix?: string;                  // 父节点 prefix, 用于子级 prefix 计算
  children: NavNode[];                     // 子节点
  is_closing?: boolean;                    // 标记为封尾节点
  default_template?: string;               // 封尾节点的默认致谢模版
}
```

### 5.2 自动编号规则

```js
function renumberPrefixes(tree, format) {
  // level 1 (chapter) 编号
  let level1Counter = 0;
  tree.forEach(node => {
    if (node.type === 'chapter' && !node.is_closing) {
      level1Counter++;
      node.prefix = format.level1.replace('{N}', level1Counter);
      // level 2 (page) 编号
      let level2Counter = 0;
      node.children.forEach(child => {
        if (child.type === 'page') {
          level2Counter++;
          child.prefix = format.level2.replace('{N}', level1Counter).replace('{M}', level2Counter);
          child.parent_prefix = node.prefix;
        }
      });
    }
  });
}
```

### 5.3 增 / 删 / 改 / 拖拽 操作

| 操作 | UI 行为 | prefix 重算 |
|---|---|---|
| 新增 chapter | 右键 "新增一级标题" → 弹命名框 → 插入到同级末尾 → prefix = 自动下一个 | ✓ 触发全局 renumber |
| 新增 page | 右键 chapter 节点 "新增子页" → 弹命名框 + 选 page_id (从 pages_data 候选) → 插入到 children 末尾 | ✓ |
| 改 name | 双击节点 → 文本框编辑 | 不触发 (prefix 不变) |
| 改 prefix | 右键 "编辑前缀" → 弹框 (默认显示自动值, 可改) | 不触发 (手动锁定) |
| 拖拽 | 拖到另一个 chapter 下 → 自动重新归类 | ✓ 触发父节点 renumber |
| 拖拽到 chapter 同级 | 重排同级顺序 | ✓ |
| 删除 | 右键 "删除" → 确认 → 从 tree 移除 | ✓ |
| 恢复默认前缀 | 右键 "恢复自动前缀" | ✓ 用 format 重算 |

### 5.4 封尾节点特殊行为 (Q3 回答)

- 系统默认有 1 个 **"封尾" 模版节点** (id="n_default_closing"), `name="封尾"`, `prefix=null`, `is_closing=true`, `default_template="致谢模版默认内容..."`
- 这个节点 **v1 默认不显示** 在侧边栏 (v3 设计就是 "你现在不要显示")
- 用户首次 **添加 1 个新的一级标题并命名为"封尾"** → 检测到命名冲突 (default_closing 已存在但未显示), 自动:
  - 把用户新节点的 `default_template` 设为预设致谢模版
  - 隐藏 default_closing 节点
- 用户编辑 content 后, 节点右键 "恢复默认模版" → 把 content 重置为 `default_template`

### 5.5 拖拽交互 (v1 用 HTML5 drag API, v2 换 dnd-kit / react-dnd)

```js
// v1 简化版
nodeEl.draggable = true;
nodeEl.addEventListener('dragstart', e => {
  e.dataTransfer.setData('text/node-id', node.id);
});
nodeEl.addEventListener('dragover', e => e.preventDefault());
nodeEl.addEventListener('drop', e => {
  e.preventDefault();
  const draggedId = e.dataTransfer.getData('text/node-id');

  moveNode(draggedId, targetId, 'before' | 'after' | 'inside');
  renumberPrefixes(nav_tree, prefix_format);
  rerender();

```

### 5.6 节点 vs page_id 的 1:1 约束

- type='page' 的节点必须有 page_id, 且 page_id 必须存在于 pages_data

### 5.7 与 v3 的差异

v3 章节 12 个 (硬编码). v5 树结构是 **数据**, 节点数 / 层级 / 命名都可改. v1 初始化树时按 v3 的 12 章节 + v5 新增 2 章节 (= 14 节点) 构造, 用户后续可增删.

## 6. 角色 × 状态机 (v5 修订)

### 6.1 角色定义

v3 定义 3 角色 (填报人/复核人/阅读人). v5 沿用, 但明确:

| 角色 | 职责 | v1 范围 | 权限 |
|---|---|---|---|
| 填报人 | 录入数据、调 AI 草稿、接受/丢弃 | **✓** | 全部章节可编辑 |
| 复核人 | 看异常、审 AI 草稿、确认导出 | **✓** (同 UI 切换) | 全部章节只读 + "✓ 通过" 按钮 |

**v1 关键约束**: 填报人可在顶部切到 "复核视角" (UI 加个 toggle), **但底层数据模型同时存两套状态**:
- `submitter_actions`: 填报人做过什么 (录入了哪些, 接受了哪些草稿)
- `reviewer_actions`: 复核人做过什么 (点了哪些 "✓ 通过", 驳回了哪些)

v2 切鉴权时, 同一 user 登录, 但 role 字段决定他看到填报人视图还是复核人视图.

### 6.2 状态机: 周报生命周期

```
[draft] --填报人提交--> [submitted] --复核人通过--> [approved] --导出--> [archived]

   |                         |                          |
   复核人驳回                 复核人撤回通过              归档
   v                         v                          v
[rejected]                [draft]                     (终态)
```

### 6.3 状态机: 单条数据 (chapter 级别)

```
[empty] --录入--> [draft] --AI 抽取--> [ai_drafted] --用户接受--> [confirmed]
   |                  |                       |                          |
   |                  |                       |                          |
   |                  用户改字段               用户丢弃                     复核人通过
   |                  v                       v                          v
   |              [draft]                 [draft]                    [approved]
   |                                                                     |
   |                                                                  复核人驳回
   |                                                                     v
   |                                                                 [draft]
   |
   用户从未录入
```

v1 持久化字段: `pages_data[id].status` (per-chapter). v1 默认都是 `draft`.

### 6.4 UI 切换角色 (v1 同 UI)
