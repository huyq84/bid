# 百草园周报填报平台 — 方案设计 v4：新建周报工作流

> 版本: v4 (基于 v3 方案 + Step 1~2 细化)
> 日期: 2026-06-05
> 上游: 方案设计_v3.md (保持不变，作为设计纲领)
> 性质: v4 是 v3 在"新建周报"主流程上的实施级细化，可直接交给 Claude Code / 前端开发照着做
> 范围: 不改 v3 已敲定的角色、强度、入口分类、规则、UI 色板；只补"流程怎么走、版式怎么分、状态怎么转"

---

## 0. 出发点：v3 没解决的三件事

把 v3 通读一遍后，实际动工前还有 3 件事没说清，v4 就是要把这 3 件事钉死：

1. "新建周报"主流程没定义清楚 — v3 只在 README 演示路径里写了 9 步散乱动作，没有"打开 → 选周期 → 复制上周 → 录入 → 异常 → 预览 → 导出"的状态图，也没有"复制上周"具体复制什么不复制什么。
2. "一版多页"机制没体现 — 5.18 那期 PDF 是 46 页，HTML 还原了 13 张版式 (01..12 + 0301)。13 张版式 × 重复次数 = 46 页，这是 platform 数据模型最关键的一层，但 v3 的 JSON 模型和章节渲染器都把每个章节当成单实例，没体现"04 这一版式在 5.18 那期被实例化 5 次（5 个施工场地）"。
3. LLM "建议" vs "自动填"的边界没定义 — v3 给了 4 种入口、7 条规则、6 个 AI 章节，但没说每个字段 LLM 是只给候选、还是直接入库、还是和上次对账。这关系到 prompt 设计和"草稿卡"长什么样。

v4 后续章节依次解决这 3 件事 + 异常状态机 + 预览→导出。

## 1. 新建周报主流程 (Step 1)

### 1.1 流程图 (文字版)

```
[填报人打开 platform/index.html]
        |
        v
[(1) 选周期]  <- 顶部下拉：本期 Wxx / 上期 Wxx-1 / 跳过
        |
        |-- 选 "本期 Wxx" (默认 W21)
        |       |
        |       v
        |   [(2) 初始化本期]
        |       |-- Store.load(Wxx) -> 已有数据? 是 -> 直接进 (3)
        |       |                                |
        |       |                                否 -> seed(Wxx) -> 走"从 Wxx-1 复制"策略
        |       |                                          |
        |       |                                          v
        |       |                                  [保存为草稿] -> 进 (3)
        |       |
        |       v
        |   [(3) 录入]
        |       |-- 6 个 AI 章节走 4 种入口 (语音/粘贴/录入/附件)
        |       |-- 7 个非 AI 章节直接录入
        |       |-- 每次保存 -> 触发 Rules.evaluate() -> 异常条幅刷新
        |       |
        |       v
        |   [(4) 异常处理]  <- 顶部条幅
        |       |-- 0 异常 -> 出现绿色 对号 -> 进 (5)
        |       |-- 蓝/黄 -> "知道了" -> 仍可进 (5)
        |       |-- 红 -> v1 暂不阻塞，但弹"是否仍要导出"确认框 -> 进 (5)
        |       |
        |       v
        |   [(5) 预览 13 张版式]
        |       |-- preview.html iframe 加载 preview_pages/01..12.html
        |       |-- 一版多页：iframe 内由版式 JS 读 report.versions[id] 渲染多次
        |       |-- 顶部翻页器 (13 -> 46) 逐页确认
        |       |
        |       v
        |   [(6) 导出]
        |       |-- 导出 PDF -> Playwright 后端批量 -> 百草园-清尚-周报YYYY.M.DD.pdf
        |       |-- 导出 Word -> v2 (按钮留)
        |       |-- 导出 PPT -> v2 (按钮留)
        |       |
        |       v
        |   [(7) 归档]  <- v2 范围
        |       |-- Store.upload(report, period) -> FastAPI -> PG JSONB
        |
        |-- 选 "上期 Wxx-1"
        |       |-- 直接 Store.load(Wxx-1) -> 只读浏览 -> 右上角"另存为本期 Wxx+1"
        |
        |-- 选 "跳过"
                |-- 顶部按钮"新建空白周报" -> 自定义周期 (v2 再做)
```

### 1.2 流程中的角色切换点

| 阶段 | 当前角色 | 可切换到 |
|---|---|---|
| (1)(2)(3) | 填报人 | 复核人 (v1 同一人，UI 切换 tab) |
| (4) | 填报人 | 复核人 (异常复核) |
| (5) | 填报人 / 复核人 | 阅读人 (v2 切到只读模式) |
| (6) | 复核人 | 填报人 (v1 同一 UI) |
| (7) | 系统 | — |

v1 暂不实现"角色切换" — 所有动作都假设是填报人。但 第 5 节 数据模型里要预留 submitter.role 和 receiver.role 字段，v2 加鉴权时不用重构数据。

### 1.3 "从 Wxx-1 复制" 策略表 (关键设计)

新周期 seed 时，不 是无脑全复制，而是按字段 × 章节的"复制语义"表：

| 章节 | 字段 | 复制语义 | 不复制 |
|---|---|---|---|
| 01 封面 | title, subtitle, project, general_contractor, subcontractor | 全部继承 | report_date, period |
| 02 目录 | 静态模板 | — | — |
| 03 组织架构 | roster[].role/name/phone | 全部继承，所有 delta 改为 new | roster[].present (清空), roster[].delta |
| 0301 里程碑 | 整个数组 | 全部继承 | — |
| 04 上周完成 | work_done | 任务文本继承，但全部 progress 重置为 0%，移到下周的 carry_from 标记 | deadline, source |
| 05 工作现场 | site_photos | 全部清空 | — |
| 06 人员统计 | labor_stats[].type, this_week=上期 next_week | 字段改名：上期 next_week -> 本期 this_week | next_week |
| 07 ECC 销项 | ecc_summary | 全部清空 (ECC 数字每周重算) | — |
| 08 图纸深化 | design_deepen | 全部清空 | — |
| 09 下周计划 | next_week_plan | 清空，由 LLM 从 04 续排 | — |
| 10 施工段划分 | construction_sections (楼层图) | 全部清空 (每周重传) | — |
| 11 施工段计划 | section_schedule | 保留，cells 全部减一格 (下推一周) | highlight |
| 12 协调事宜 | coordination | 清空 (每期协调事项不同) | — |

v1 实施细节：
- 在 Store.seed(period) 里增加一个 prev = Store.load(prevPeriod) 参数，按上表构造本期 seed。
- 如果 prev 为 null (系统里只有本期一期)，用 DEFAULT_PERIODS 里的 W20 兜底。
- 在 UI 上把"复制策略"做成一个侧边栏对照表 (v1 简化为 read-only 提示)，让填报人知道"我打开本期时哪些字段是空的"。

### 1.4 周期选择的"本期/上期"判定

- 默认 currentPeriod = W21 (最新一周)。
- 顶部下拉只显示存在数据的周期 (Store.getReportList())，按时间倒序。
- "上期" = 当前周期的前一周 (按 ISO week 算)。
- v1 暂不实现"自定义周期"。v2 在顶部加"新建空白周报"按钮。

## 2. 一版多页模型 (Step 2 — 架构最关键)

### 2.1 核心问题

5.18 那期 46 页 = 13 个版式 × 各自的实例数。具体：

| 版式编号 | 5.18 期页数 | 实例化依据 | 5.18 实际 |
|---|---|---|---|
| 01 封面 | 1 | 固定 | 1 |
| 02 目录 | 1 | 固定 | 1 |
| 03 组织架构 | 1 | 固定 | 1 |
| 0301 里程碑 | 1 | 固定 | 1 |
| 04 上周完成 | 5 | 5 个施工场地 (高管区/食堂区/南塔健身/北塔健身/南北咖啡) | 5 |
| 05 工作现场 | 5 | 同 4.2 的 5 个场地 | 5 |
| 06 人员统计 | 1 | 固定 | 1 |
| 07 ECC 销项 | 1 | 固定 | 1 |
| 08 图纸深化 | 1 | 固定 | 1 |
| 09 周计划 | 1 | 固定 | 1 |
| 10 施工段划分 | 3 | 食堂 B1/一层/二层 | 3 |
| 11 施工段计划 | 1 | 固定 | 1 |
| 12 协调事宜 | 1 | 固定 | 1 |
| 小计 | 23 页 | | — |
| 实际 PDF | 46 页 | | 多出的是大表分页 + 页眉/页脚重复 |

> 注：实际 46 = 23 版式页 + 13 重复实例的"溢出页" (4.2 高管区单页装不下 9 行，第二页继续画)。这部分溢出由版式 JS 内部处理，不算独立版式。

### 2.2 三层模型：版式 × 实例 × 数据

```
[版式 Template]   1 --N--> [实例 Instance]   1 --N--> [数据行 DataRow]
   04                  5                 9
   05                  5                 6 张照片
   10                  3                 1 张楼层图
```

TypeScript/JSON 描述 (伪代码，便于 v1 落地)：

```ts
// 版式定义 (编译期 / JSON Schema 都行)
type Template = {
  id: "04" | "05" | "06" | ... | "12"
  type: "single" | "repeatable"        // 04/05/10 是 repeatable
  repeatBy?: "area" | "floor" | "section"
  repeatValues?: string[]               // 实例化时的取值范围
  dataShape: "table" | "photo" | "image" | "form"
  fields: FieldSpec[]                   // 见第 3 节 LLM 矩阵
}

// 5.18 那期版式清单
const TEMPLATES: Template[] = [
  { id: "04", type: "repeatable", repeatBy: "area", dataShape: "table" },
  { id: "05", type: "repeatable", repeatBy: "area", dataShape: "photo" },
  { id: "06", type: "single",     dataShape: "table" },
  { id: "07", type: "single",     dataShape: "table" },
  { id: "08", type: "single",     dataShape: "table" },
  { id: "09", type: "single",     dataShape: "table" },   // 内部多行
  { id: "10", type: "repeatable", repeatBy: "floor", repeatValues: ["一层","二层","B1"], dataShape: "image" },
  { id: "11", type: "single",     dataShape: "table" },
  { id: "12", type: "single",     dataShape: "table" }
]
```

### 2.3 数据模型扩展 (与 v3 JSON 模型差异)

v3 的 JSON 模型把每个章节当数组：

```jsonc
// v3 模型 (节选)
"work_done": [
  {"area":"高管区","seq":1,"task":"...","progress":"100%","owner":"鲍永春"},
  {"area":"食堂区","seq":1,"task":"...","progress":"100%","owner":"..."}
]
```

v4 改成版式 × 实例 × 数据 三层：

```jsonc
// v4 模型 (节选)
"templates": {
  "04": {
    "type": "repeatable",
    "repeatBy": "area",
    "instances": [
      {
        "id": "04-gaoguan",        // 实例 id = 版式编号-取值
        "sub_title": "4.2 高管区",
        "rows": [                  // 数据行
          {"seq":1,"task":"一层高管办公室龙骨吊顶","progress":"100%","owner":"鲍永春","deadline":"2026-05-17","source":"voice"},
          {"seq":2,"task":"二层房间内地暖盘管","progress":"100%","owner":"鲍永春",...}
        ]
      },
      {
        "id": "04-shitang",
        "sub_title": "4.2 食堂区",
        "rows": [...]
      }
      // ... 共 5 个实例 (按 v1 实际填报时确定)
    ]
  },
  "05": {
    "type": "repeatable",
    "repeatBy": "area",
    "instances": [
      { "id":"05-gaoguan", "sub_title":"2.2 高管层现场工作", "photos":["p1.jpg","p2.jpg","p3.jpg","p4.jpg","p5.jpg","p6.jpg"] }
      // ... 共 5 个实例
    ]
  },
  "10": {
    "type": "repeatable",
    "repeatBy": "floor",
    "instances": [
      { "id":"10-f1", "sub_title":"4.1 食堂区施工段划分图 - 一层", "image":"floor1.png" },
      { "id":"10-f2", "sub_title":"4.1 食堂区施工段划分图 - 二层", "image":"floor2.png" },
      { "id":"10-b1", "sub_title":"4.1 食堂区施工段划分图 - B1层", "image":"b1.png" }
    ]
  },
  "06": { "type":"single", "data": { ...原 v3 labor_stats 数组 } },
  // ...
}
```

关键设计点：
- 实例是平铺的 (不是 instances: [{id, sub_title, rows: [...]}] 套娃)，因为有些实例的 data 是表格 (rows)，有些是图片 (image)，有些是表单 (data)。
- sub_title 是实例级字段，不是版式级 — 因为 04 的"4.2 高管区"和"4.2 食堂区"是同一个版式但不同实例的副标题。
- repeatBy 是枚举：v1 只支持 area / floor / section，v2 可扩。

### 2.4 渲染管线

preview.html 当前是 iframe src="preview_pages/01.html"，单实例。

v4 改成：

```js
// preview.html 里
function renderTemplate(template, instance) {
  // 方案 A: iframe 仍然只加载版式 HTML，但通过 postMessage 注入实例数据
  iframe.contentWindow.postMessage({ template, instance }, '*')
  // 方案 B: 版式 HTML 升级为可读 report JSON 自行渲染 (更彻底)
}
```

v1 建议采用方案 A：
- 改动最小，13 张版式 HTML 几乎不动 (只增加 1 个 <script> 接收消息)。
- 已经在 preview_pages/ 里有镜像文件，直接复用。

## 3. 字段级 LLM 介入矩阵 (Step 3 节选)

v3 给了 4 种入口、6 个 AI 章节，但没说每个字段 LLM 是建议/抽取/自动填。下面这张表是 v4 给出的细分。

| 章节 | 字段 | LLM 行为 | 置信度阈值 | 草稿卡行为 |
|---|---|---|---|---|
| 04 上周完成 | area | 抽取 (从文本识别场地) | 0.7 | 草稿卡可改 |
| 04 | task | 抽取 (主事项) | 0.9 | 草稿卡可改 |
| 04 | progress | 抽取 ("完成"/数字) | 0.8 | 草稿卡可改 |
| 04 | owner | 抽取 (人名匹配 OWNER_DICT) | 0.95 | < 0.95 高亮黄 |
| 04 | deadline | 抽取 (日期识别) | 0.7 | < 0.7 高亮黄 |
| 04 | source | 自动填 (voice/paste/file) | — | 不显示 |
| 05 工作现场 | photos | AI 分组 (按 area 自动归到 5 个实例) | 0.6 | 草稿卡可拖拽改 |
| 06 人员统计 | labor_stats[].type | 抽取 (工种识别) | 0.9 | 草稿卡可改 |
| 06 | this_week / next_week | 对账 (vs 花名册签到) | 0.8 | < 0.8 标 警告 |
| 07 ECC 销项 | total/closed/in_process/open | OCR 抽取 (从截图/Excel) | 0.85 | < 0.85 标 警告 |
| 07 | close_rate | 自动算 (derived) | — | 不显示 |
| 08 图纸深化 | task/owner/status | 抽取 (从语音/粘贴) | 0.85 | 草稿卡可改 |
| 09 下周计划 | area/task/days | AI 排甘特 (从 04 carry + 新增) | 0.7 | 草稿卡可拖动改 |
| 09 | progress[7] | 自动生成 (按 days 推算) | — | 草稿卡可点击改 |
| 09 | labor/material | 抽取 (从知识库) | 0.6 | < 0.6 标 警告 |
| 11 施工段计划 | cells[12] | 续排 (基于 11 上一期下推一周) | — | 草稿卡可改 |

"草稿卡行为"列是关键：
- 绿 (>= 0.95)：直接入库，不显示草稿卡。
- 黄 (0.7~0.95)：草稿卡可"接受 / 改" — v1 行为已实现。
- 红 (< 0.7)：草稿卡标红，必须人工改。

v1 的 llm.js 已实现 04 的 _confidence 字段 (草稿卡置信度色条)，但只对 04 做了。v4 把这套机制推广到 06/07/08/09/11。

### 3.1 4 种入口的字段支持差异

| 入口 | 支持字段 (v1) | 不支持 |
|---|---|---|
| 语音 | 04 task/owner/progress/deadline；08 task/owner/status；09 task/area | 表格类 (07 ECC、11 计划) |
| 粘贴 | 04/08/09 同上；06 工种识别 | 11 施工段计划 (需对账) |
| 表单 | 全部字段 | — |
| 附件 | 06 花名册 Excel；07 ECC 截图/Excel；05 照片 zip | 09 暂不支持附件 |

## 4. 异常状态机 (Step 4)

v3 给了 3 级颜色 + 7 条规则，但没定义状态怎么转。v4 明确：

```
[idle] --录入保存--> [evaluating] --规则命中--> [anomaly_open]
                                              |
                                          用户编辑该字段
                                              |
                                              v
                                        [evaluating] (re-render 时跑)
                                              |
                                          规则不再命中
                                              |
                                              v
                                         [anomaly_resolved]
                                              |
                                          1 次重 render 后从条幅消失
```

关键设计点：

1. 异常状态是 derived，不是持久。v3 当前实现已经是这样 (Rules.detectAll 每次重跑)，v4 沿用并明确。
2. 3 级异常的"阻塞"行为：
   - 红：v1 弹"是否仍要导出"确认框 (v3 说"v1 暂不阻塞"，v4 同此)。v2 改为硬阻塞。
   - 黄：UI 提示，不阻塞。
   - 蓝：UI 提示，不阻塞。
3. "重新生成 AI 草稿"按钮 (顶部 齿轮)：调 rebuildAI() -> 重新跑 Rules.detectAll + 重新调 LLM 生成所有草稿卡 (v1 行为)。v4 明确：此按钮也清空所有"已接受"的草稿 — 避免草稿叠加。
4. "carry-over-untouched" 规则语义修正 (v3 文档与代码不一致)：
   - v3 README 写"上周进行中未续排"为黄 — 隐含语义是"未续排=异常"。
   - 当前 rules.js 实现：continued_from_last_week.length > 0 就报黄 — 这是反向语义，"已续排"也报黄。
   - v4 修正：check 应为"上周进行中 ∩ 本周 09 计划 ∩ 仍含此任务" 才报黄，否则视为已续排 (正常)。具体见第 6 节 fact-check 清单。

## 5. 预览 -> 导出 (Step 5 概览)

### 5.1 预览

preview.html 当前：顶部按钮栏 13 个 + iframe 1 个。

v4 改成：
- 顶部按钮栏从 13 改成 23 (按 第 2.5 节 实例化数)，按版式编号 + 实例 id 命名 (如 04-gaoguan)。
- iframe 加载逻辑：按当前选中实例动态切 src。
- 支持键盘 左/右 翻页。

### 5.2 导出 PDF

v3 留空。v4 给出具体技术选型：

| 选型 | 优劣 | v1 建议 |
|---|---|---|
| 浏览器 window.print() | 简单，但只能 1 页 | 临时方案 |
| Playwright 后端批量 | 可靠、可复现、与 platform 解耦 | v1 用这个 |
| pdfkit / puppeteer 原生 | 灵活但工程量大 | v2 |

v1 实施步骤：

## 11. Fact-Check：v4 写作中发现的事实问题

> 写 v4 时我反复对照了 v3 文档 vs 实际代码 (store.js / rules.js / llm.js / app.js / sections/*.js)，
> 下面这些点是**文档与代码不一致**或**代码有 bug** 的地方，需要你拍板再继续：

### 11.1 `carry-over-untouched` 规则语义反了

**v3 README** 描述：
> "carry-over-untouched | 🟡 黄 | 上周进行中未续排"

**实际 `rules.js` 代码**：
```js
check: (r) => (r.ai_review?.evidence?.continued_from_last_week || []).length > 0
```
**结果**：只要 `continued_from_last_week` 数组非空就报黄。也就是"**已续排**"也报黄，跟 README 写的"**未续排**"是反的。

**v4 修正建议**：
- 真正的"未续排"应该是：上周 `work_done[]` 里 `progress < 100%` 的任务，**且**本周 09 `next_week_plan[]` 里**没有**对应 task 文本。
- 实施成本：中等（需要文本相似度匹配）。v1 简化版：按 `area + owner` 精确匹配。
- **建议 v1 直接改成"已续排=绿/不报，未续排=黄"**，与 README 描述对齐。

### 11.2 `.env` 末尾行被错拼了 MockJS CDN 链接

**实际 `.env` 内容末尾**：
```
REDIS_URL=redis://localhost:***@1.1.0/dist/mock-min.js
```
明显是把 Redis 端口和 `https://cdn.jsdelivr.net/npm/mockjs@1.1.0/dist/mock-min.js` 拼到一行了。

**风险**：如果有人 `source .env` 然后让代码 `urllib.parse.urlparse(os.environ['REDIS_URL']).port` 之类的，会读到一个错的端口。

**v4 建议**：
- v1 阶段 `.env` 不用（纯前端，没后端）—— 直接删掉。
- v2 阶段重建 `.env.example`，分清楚变量。

### 11.3 `seed()` 凭空生成，没有"从上周复制"

**实际 `Store.load(period)` 行为**：
```js
const s = localStorage.getItem(key);
if (s) { return JSON.parse(s) }
// 否则
return seed(period);  // 凭空生成
```

**结果**：5.19（周一）打开 W21 期，看到的是 `seed()` 默认数据，**与 W20 期完全无关**。这是 v1 最大的设计缺口。

**v4 修正**：见第 1.3 节策略表，已在 v4 给出"哪些字段继承、哪些清空"的逐字段表。

### 11.4 章节渲染器全单实例

**实际 `sections/s04_work_done.js`**：
```js
render(report) {
  const items = report.work_done.map((t, i) => `<tr>...`).join("");
  // 只渲染 report.work_done 数组一次
}
```

**问题**：v3 的 `work_done[]` 是 9 行混合了 1 个 area 的数据。但 5.18 那期 `work_done` 实际是**5 个 area × 9 行** = 45 行分散到 5 张版式上。**v3 没说这 5 张怎么实例化**。

**v4 修正**：见第 2 节"一版多页模型"。`templates.04.instances[]` 数组，**每个 instance 是 1 个 area 的所有 rows**。

### 11.5 `s05_photos.js` 写死 "高管层"

**实际代码**：
```js
const photos = report.site_photos["高管层"] || [];
```

**问题**：v3 数据模型里 `site_photos` 是 `{"高管层": [...6 张图]}` 单 key 结构。但 5.18 那期应该有 5 个 area 各 6 张图。

**v4 修正**：改成 `report.templates["05"].instances[]`，每个 instance 有 `photos: []` 数组。

### 11.6 `aiGroupPhotos` / `aiExtendSchedule` 是 stub

**实际 `app.js`**：
```js
function aiGroupPhotos() { toast("AI 照片分组功能开发中", "info"); }
function aiExtendSchedule() { toast("AI 扩展进度计划功能开发中", "info"); }
```

**问题**：v3 文档第 2 节"章节与填报强度" 表格里写着 05 / 11 是"AI 自动化"，但**实现是 toast 提示**。

**v4 修正**：v1 Sprint 2 实施时把这两个函数补完。`aiGroupPhotos` 走 05 的 mock 照片分类（按上传顺序/EXIF 时间分到 5 个 area）；`aiExtendSchedule` 走 11 的 mock 续排（基于上期 cells 下推一周）。

### 11.7 顶部按钮栏 13 个 vs 实际 23 个

**实际 `preview.html`**：
```js
const PAGES = [ {id:"01",...}, {id:"02",...}, ..., {id:"12",...} ];  // 13 个
```

**问题**：5.18 那期 46 页 = 13 个版式 + 33 个重复实例。当前 preview 只覆盖了 13 张。

**v4 修正**：见第 5.1 节，按"版式编号-实例 id"重命名 23 个按钮。

### 11.8 `SeedPeriod` 与 store.js 默认不一致

**实际 `store.js`**：
```js
const DEFAULT_PERIODS = ["2026-W18", "2026-W19", "2026-W20", "2026-W21", "2026-W22"];
const DEFAULT_CUSTOM_PERIODS = {
  "2026-W18": { start: "2026-04-28", end: "2026-05-04", label: "第18周" },
  ...
}
```

**问题**：DEFAULT_PERIODS 与 DEFAULT_CUSTOM_PERIODS 都硬编码，**改一边要同步另一边**。v3 数据模型说"period_config 来自后端"，但 v1 是 localStorage 硬编码。

**v4 建议**：v1 把这两份合并到一个 `PERIODS_CONFIG` 常量，v2 切后端时只换数据源。

### 11.9 README/CLAUDE.md 与实际不一致

- 根目录 `CLAUDE.md` 没提 `platform/` 存在 → 新人会漏看。
- `platform/README.md` 描述的"7 条异常规则" 与 `rules.js` 实际条数一致 ✓ 但 `carry-over-untouched` 描述与代码语义反（见 11.1）。
- README 说"13 个章节全部渲染" ✓，但**没说一版多页**（实际 5.18 那期是 23 个实例）。

**v4 建议**：
- 根目录 CLAUDE.md 加一段 "platform/ 是工程主体，根目录 HTML 是 platform 用的版式参考"
- platform/README.md 在"v1 已覆盖"列表里加一条"一版多页 (N 个版式 × M 实例 = PDF 总页数)"
